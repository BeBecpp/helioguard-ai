# HelioGuard AI — Research Document

**NASA Space Risk Intelligence Dashboard**  
*Educational Research Proposal · Hack Club Stardance*

---

## Abstract

HelioGuard AI is an educational web dashboard that transforms NASA public near-Earth object (NEO) and space weather data into interpretable risk scores and plain-English mission briefings. The system applies transparent mathematical models — not opaque machine learning — to NeoWs asteroid feed data and DONKI space weather events, producing a composite **HelioGuard Index** (0–100) suitable for student researchers, science educators, and hobbyist space enthusiasts. This document describes the problem, methodology, formulas, architecture, limitations, and evaluation plan.

---

## Problem Statement

NASA publishes rich open data on asteroid close approaches and solar activity through NeoWs and DONKI APIs. However, raw JSON responses are difficult for non-experts — especially high school students — to interpret quickly. Existing tools either require domain expertise or present data without contextual risk framing.

There is a need for an **educational**, **transparent**, and **visually accessible** dashboard that:

1. Uses real NASA data
2. Applies explainable risk scoring
3. Does not claim official NASA endorsement
4. Does not function as an emergency alert system

---

## Research Question

**Can NASA public asteroid and space weather data be combined into a single, interpretable risk index that helps non-experts understand near-term space environment conditions?**

---

## Hypothesis

A weighted composite index combining normalized asteroid proximity factors (diameter, velocity, miss distance, hazardous classification) and space weather event severity will produce actionable, educational risk summaries that correlate with expert intuition — i.e., larger/closer PHAs and major CMEs/flares yield higher scores, while distant small NEOs and quiet solar periods yield lower scores.

## Mathematical Model

### Asteroid Risk Formula

For each near-Earth object:

```
AsteroidRisk = 100 × clamp(0.30×S_d + 0.20×S_v + 0.35×S_dist + 0.15×H, 0, 1)
```

Where:

| Symbol | Definition | Normalization |
|--------|------------|---------------|
| S_d | Diameter score | `normalizeRange(avgDiameterMeters, 0, 1000)` |
| S_v | Velocity score | `normalizeRange(velocityKmS, 0, 40)` |
| S_dist | Closeness score | `inverseNormalizeRange(missDistanceLD, 0, 20)` |
| H | Hazardous flag | 1 if `is_potentially_hazardous_asteroid`, else 0 |

**Aggregate asteroid risk** for the HelioGuard Index uses the average of the top 3 highest-scoring NEOs in the feed window.

### Space Weather Risk Formula

Individual event severity is assigned by event type (CME > FLR > GST > RBE) with text-based boosts for strong classifications (e.g., X-class flares).

```
SpaceWeatherRisk = 0.6 × max(severities) + 0.4 × avg(top 3 severities)
```

**Fallback:** When DONKI endpoints fail, a conservative baseline score of **8** is applied so the dashboard remains functional.

### HelioGuard Index Formula

```
HelioGuardIndex = 0.45 × AsteroidRisk + 0.55 × SpaceWeatherRisk
```

Space weather is weighted slightly higher because solar events can affect global technology infrastructure (satellites, HF radio, power grids) even when no asteroids pose immediate concern.

### Risk Level Thresholds

| Score Range | Level |
|-------------|-------|
| 0 – 24 | LOW |
| 25 – 49 | MODERATE |
| 50 – 74 | ELEVATED |
| 75 – 100 | HIGH |

## AI-Style Briefing Engine

The mission briefing is generated by a **rule-based template engine** (`briefingEngine.ts`), not a large language model. Inputs:

- HelioGuard Index and level
- Asteroid and space weather sub-scores
- Top-ranked asteroid metadata
- DONKI event count and availability status

Output sections:

1. Index summary
2. Top asteroid highlight
3. Space weather status
4. Technology impact guidance (scaled by level)
5. Educational disclaimer

**Rationale:** Rule-based generation is reproducible, auditable, and appropriate for an educational MVP without paid AI API costs.

## System Architecture

```
Browser (React + Vite)
    │
    ├── nasaNeoService.ts ──► NASA NeoWs Feed
    │       └── asteroidRisk.ts (per-NEO scoring)
    │
    ├── nasaDonkiService.ts ──► NASA DONKI (CME, FLR, notifications)
    │       └── spaceWeatherRisk.ts (event severity)
    │
    ├── briefingEngine.ts ──► Mission Briefing text
    │
    └── Components
            ├── Hero.tsx
            ├── RiskIndexCard.tsx
            ├── AsteroidCard.tsx
            ├── SpaceWeatherTimeline.tsx
            ├── MissionBriefing.tsx
            └── DataSourcePanel.tsx
```

**Design principles:**

- Thin services, fat utilities (testable math)
- Graceful degradation when DONKI fails
- TypeScript interfaces for all NASA payloads
- No backend — suitable for static deployment

## Limitations

1. **Not predictive:** Risk scores are heuristic, not orbital mechanics simulations.
2. **7-day window:** NeoWs feed covers a short horizon only.
3. **DEMO_KEY rate limits:** NASA demo key throttles requests; production use needs a personal key.
4. **No historical analysis:** MVP shows current window only.
5. **Simplified space weather model:** Full geomagnetic storm (GST) integration is planned for future work.
6. **Educational only:** Must not replace NASA, NOAA, or government alert systems.

---

## Ethics and Safety

- **No false authority:** The dashboard clearly states it is educational and not an official NASA product.
- **No panic framing:** Language emphasizes monitoring and official sources, not alarm.
- **Transparent math:** All formulas are documented and implemented in readable utility functions.
- **Open data attribution:** NASA API sources are linked in the UI and documentation.
- **No personal data:** MVP collects no user PII.
